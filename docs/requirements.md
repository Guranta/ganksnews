# LittleGankNews 需求文档

## 项目概述

构建一个针对 Twitter/X 公开账号的自动化监控系统，通过 CloakBrowser / Playwright 持久化浏览器 Profile 监听公开页面，实现推文采集、处理与告警。不使用 X API v2，也不使用 snscrape。

---

## 一、功能需求

### 1.1 数据采集

| 需求编号 | 功能描述 | 优先级 | 备注 |
|---------|---------|--------|------|
| F-001 | 监控指定 Twitter/X 公开用户的新推文发布 | P0 | 核心功能，Phase 1 先 20-30 个账号 |
| F-002 | 采集推文内容（文本、图片、视频链接） | P0 | 需包含多媒体信息 |
| F-003 | 采集推文元数据（发布时间、点赞数、转发数、评论数） | P0 | 用于分析用户影响力 |
| F-004 | 采集用户资料变更（头像、简介、昵称等） | P1 | 可选 |
| F-005 | 支持增量采集，避免重复数据 | P0 | tweet_id 去重 + 数据库唯一约束 |
| F-006 | 支持目标账号批量导入（纯文本一行一个 username） | P0 | Phase 1 必做 |
| F-007 | 保存解析失败的截图、HTML、错误栈 artifacts | P0 | 排查和 parser 版本化 |

## 二、非功能需求

### 2.1 性能需求

| 需求编号 | 指标 | 目标值 | 备注 |
|---------|------|--------|------|
| NF-001 | 数据采集延迟 | 尽量快，不承诺 < 1 分钟 | 浏览器监听方式受页面刷新频率限制 |
| NF-002 | 系统可用性 | Worker 崩溃后可重启 | 不追求 99.5%，先保障可恢复 |
| NF-003 | Phase 1 并发能力 | 1 个 Profile 监听 20-30 个账号 | 后续可扩展到 500+ |
| NF-004 | Artifacts 磁盘控制 | 保留天数限制 | 避免磁盘爆满 |

### 2.2 可靠性需求

| 需求编号 | 功能描述 | 优先级 |
|---------|---------|--------|
| NF-005 | Worker 心跳与健康检查 | P0 |
| NF-006 | 数据至少一次投递（Redis Streams + ACK） | P0 |
| NF-007 | 失败消息进入 dead letter | P0 |
| NF-008 | 登录态失效检测与标记 | P0 |
| NF-009 | 解析失败保留 artifacts | P0 |

### 2.3 安全与合规

| 需求编号 | 功能描述 | 优先级 | 备注 |
|---------|---------|--------|------|
| NF-010 | 不破解验证码，不绕过平台风控 | P0 | 标记 challenged，人工处理 |
| NF-011 | 不采集私信、非公开账号或受限内容 | P0 | 只监听公开页面 |
| NF-012 | Artifacts 可能含敏感信息，限制目录权限 | P1 | 不提交 git |
| NF-013 | Phase 1 不做鉴权（仅内网或本地访问） | P1 | 公网部署时需加鉴权 |

### 2.4 可扩展性

| 需求编号 | 功能描述 | 优先级 |
|---------|---------|--------|
| NF-014 | 支持从 20-30 扩展到 100、500 个目标账号 | P1 |
| NF-015 | 支持多 Profile 并发监听 | P1 |
| NF-016 | 模块化设计，便于功能扩展 | P1 |

## 四、用户场景

### 场景 1：Crypto/Web3 信息发现
> 监控 20-30 个 KOL/项目方账号，及时发现新推文，SSE 实时推送到网页端。

### 场景 2：品牌舆情监控
> 监控 KOL/媒体账号，及时发现品牌负面提及，快速响应。

### 场景 3：竞品动态追踪
> 监控竞品官方账号和核心员工账号，获取产品发布、战略动向等信息。

## 五、Phase 1 边界

### 做什么

- 监控 20-30 个公开 Twitter/X 目标账号。
- 支持目标账号批量导入。
- 支持监听账号和 Browser Profile 的网页端登记、暂停、恢复、状态查看。
- 支持推文去重、入库、查询、搜索基础能力。
- 支持 Latest Tweets / 最新推文页面展示已入库推文，并通过 SSE 实时插入新推文。
- 支持 Worker 心跳、Redis Stream 队列状态、错误证据查看。
- 支持网页端 SSE 实时通知。
- 预留 Telegram 表结构和接口扩展点，但 Phase 1 不实现。

### 不做什么

- 不使用 X API v2。
- 不使用 snscrape。
- 不破解验证码。
- 不绕过平台风控。
- 不采集私信、非公开账号或受限内容。
- Phase 1 不做关键词规则、情感分析、异常检测、多租户、复杂 RBAC。
- Phase 1 不做 Elasticsearch / OpenSearch。
- Phase 1 不做 noVNC 服务器登录界面；Phase 2B 规划为后续服务器浏览器登录能力。
- Phase 1 不做 Prometheus / Grafana。
